Implementation Approaches

How should we architect the calling system?

A

Extend Existing Services (Recommended)

Build on CallManager + WebRTCService. Add VoiceRoomManager that reuses WebRTCService for mesh connections.

Pros

  • Leverages proven patterns
  • Less code to write
  • Consistent with codebase
  • Easier to maintain

Cons

  • WebRTCService needs refactoring for multi-peer
  • Some coupling between features
B

Separate Services Per Feature

Keep CallManager for 1-on-1. Create entirely new VoiceRoomManager with its own WebRTC handling.

Pros

  • Clean separation
  • No risk of breaking existing calls
  • Independent testing

Cons

  • Code duplication
  • Two WebRTC implementations to maintain
  • Inconsistent patterns
C

Unified Real-Time Service

Create single RealTimeCallService that abstracts both 1-on-1 and voice rooms behind unified interface.

Pros

  • Single source of truth
  • Consistent API
  • Future-proof for video rooms

Cons

  • Larger refactor needed
  • More complex abstraction
  • Higher risk of bugs
