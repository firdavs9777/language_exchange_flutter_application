# Tajik (tg) locale — full support

**Date:** 2026-05-25
**Status:** Approved, pending implementation plan
**Origin:** P4 item from the "Verification failed" registration-bug investigation guide. The Tajik user (nozil, +992) asked for Tajik support and selected Persian as a workaround. Adding `tg` is a standalone feature, decoupled from the verification bug fix.

## Goal

Ship complete Tajik (Cyrillic, `tg-TJ`) support across the BananaTalk Flutter app and its backend notification templates in a single PR. This covers two distinct surfaces:

1. **App UI language** — users who select Tajik in Settings → Language (or whose device locale is `tg`) see a fully translated app interface and receive Tajik push/email notifications.
2. **Content language** — Tajik appears as a first-class option in every place users tag or filter content by language: moments filter, moments creation, vocabulary cards, and moment-display flag chips. Tajik is already wired in `TranslationService.supportedLanguages` and the backend-driven AI/Tutor language picker, so no work is required there.

## Non-goals

- Tajik in Perso-Arabic script (`tg-Arab`). Cyrillic only.
- Web frontend (`../front`) — out of scope per the [Web frontend scope](../../../../../.claude/projects/-Users-firdavsmutalipov-Projects-BananaTalk/memory/web_scope.md) memory.
- Tajik-specific RTL/script changes — Cyrillic is LTR, no layout work needed.
- Native Tajik reviewer sign-off before merge. The user has explicitly accepted shipping an AI-drafted translation, overriding the original brief's MT warning. Post-merge corrections from real users / community translators are the expected feedback loop.

## Translation policy

- **Source of truth:** `app_en.arb` key set and any ICU placeholders (`{name}`, `{count}`, plural blocks) must be preserved verbatim — translate around them, never replace them.
- **Plural blocks:** Tajik plural rules are simple — `=0`, `=1`, `other` cover the cases. Use the same plural structure as English (`{count, plural, ...}`); Tajik does not need extra `few`/`many` categories.
- **`@`-prefixed metadata blocks:** Drop from `app_tg.arb`. Only the template (`app_en.arb`) carries `@key` descriptor blocks; locale files just carry key→value pairs. This matches the existing pattern in `app_ko.arb`, `app_ja.arb`, etc.
- **Proper nouns:** `Bananatalk`, brand names, and feature names stay in Latin script. Do not transliterate `Bananatalk` → `Бананаталк`.
- **Borrowed words:** Modern Tajik UI freely borrows from Russian (`Парол`, `Профил`, `Танзимот`) — use them where they're the natural choice rather than forcing native coinages.
- **Register:** Polite-imperative for buttons/actions (`Ворид шавед`, `Захира кунед`), descriptive for labels (`Парол`, `Забон`). Match the tone of existing locales like `ru` and `fa` for cross-locale consistency.
- **Errors:** Phrase errors in a calm, non-accusatory tone. Avoid passive-aggressive copy.

## Codegen step

After `app_tg.arb` lands:

```
flutter pub get          # triggers flutter gen-l10n via the l10n.yaml at lib/l10n/l10n.yaml
```

This regenerates `lib/l10n/app_localizations.dart` (adding `Locale('tg')` to the const list) and produces a new `lib/l10n/app_localizations_tg.dart`. Both regenerated files are committed (matching the repo's existing pattern of committing gen-l10n output).

If `flutter pub get` does not trigger codegen automatically:

```
flutter gen-l10n
```

## Fallback behavior

`flutter_localizations` already handles missing-key fallback to the template locale. Because we ship a complete `app_tg.arb`, no key should miss; if any key is accidentally omitted, the user sees the English string for that key — graceful degradation, no crash.

For device locales that aren't an exact match (e.g., `tg-Cyrl-TJ`, `tg-Arab-TJ`), Flutter's locale resolution falls back to the closest match. Our `Locale('tg', 'TJ')` will match `tg-TJ` and bare `tg`. A user with `tg-Arab` device locale will *not* match `tg-TJ` exactly — they will fall through to the app's default (English). This is acceptable; we're explicitly Cyrillic-only.

## Backend notification template

`backend/notification_templates/en.json` is 54 lines (small). Translate all keys into Cyrillic Tajik with the same translation policy as the ARB. Preserve any `{{placeholder}}` or `{name}` substitution markers verbatim.

Notification copy is short and high-visibility — getting it wrong is more painful than a missing UI string. Still no flag/gate per the user's "don't make it complicated" directive; rely on user reports + a follow-up PR to correct any bad strings.

## Testing & verification

Manual:
1. Run the app on simulator with `flutter run`.
2. In-app: Settings → Language → Тоҷикӣ. Verify the picker shows the native name and the Tajik flag.
3. Spot-check 6-8 screens: Login, Sign-up, Email verification, Home tab, Messages tab, Profile tab, Settings, a notifications list.
4. Trigger one backend notification (e.g., a chat message) and verify the push body renders in Cyrillic.

Automated:
- No new unit tests required. Existing localization tests (if any) should pass after regen.
- `flutter analyze` must pass — gen-l10n output should be clean.

Translation correctness is **explicitly not gated** on a native reviewer per user instruction. Expected feedback loop: real Tajik users (nozil et al.) report awkward strings → follow-up PRs.

## Out of scope / follow-up work

- Native reviewer pass on the AI-drafted translation. Track as a separate ticket; not blocking merge.
- Tajik in Perso-Arabic script — separate locale (`tg-Arab` or `fa-TJ`), separate effort.
- Backend admin UI / dashboard translations — not part of the user-facing notification template scope.
- Push-notification topic/segment configuration for `tg` users in the FCM backend — flag if this matters for delivery targeting; otherwise default audience handles it.
- Backend `languages` collection — if the AI/Tutor flows require a `tg` row in the backend's languages collection and it's missing, that's a one-row data migration in a follow-up PR. Verified during testing.

## Risks

- **Translation quality.** AI-drafted Tajik for ~1725 strings will have errors. Mitigated only by user reports post-merge — accepted by the user.
- **ICU placeholder corruption.** Easy to break `{name}` or plural blocks during translation. Mitigation: lint by diffing the placeholder set in each value before/after; manually inspect any value containing `{` for syntactic integrity.
- **Codegen failure.** A malformed ARB will fail `flutter gen-l10n` with a parse error. Mitigation: validate JSON well-formedness before commit; if codegen errors, fix the offending key before proceeding.
- **Generated `app_localizations_tg.dart` size.** Other locale files run 130-175 KB. Tajik will be similar — acceptable, no special action.
